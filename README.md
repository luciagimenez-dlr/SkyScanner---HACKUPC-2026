# WanderLens By SKYSCANNER

AI-powered travel experience platform built for the Skyscanner Hackathon.  
WanderLens helps travellers define their profile, share real experiences, and receive fully personalised itineraries generated by Gemini AI — complete with interactive maps, flight search, and community-contributed VR walkthroughs.

## Requirements

- **Python 3.12** — other versions are not guaranteed to work
- **pip** — bundled with Python 3.12
- **ffmpeg** — required for VR side-by-side video conversion
  - macOS: `brew install ffmpeg`
  - Ubuntu / Debian: `sudo apt install ffmpeg`
  - Windows: download from [ffmpeg.org](https://ffmpeg.org/download.html) and add to PATH
- A modern browser (Chrome, Firefox, Safari, Edge)

## Installation

```bash
# 1. Clone or unzip the project
cd wanderlens

# 2. Recommended: create a virtual environment
python3.12 -m venv venv
source venv/bin/activate        # macOS / Linux
venv\Scripts\activate           # Windows

# 3. Install all Python dependencies
pip install -r requirements.txt
```

## Configuration

Copy the example environment file and fill in your credentials:

```bash
cp .env.example .env
```

Open `.env` and configure the following variables:

```env
# Gemini AI — required for itinerary generation
# Obtain at: https://aistudio.google.com/app/apikey
GEMINI_API_KEY=your_gemini_key_here

# Skyscanner Indicative Search API — required for live flight data
# Obtain at: https://developers.skyscanner.net
# If not set, the application falls back to realistic mock data automatically
SKYSCANNER_API_KEY=your_skyscanner_key_here

# Set to true to force mock flight data regardless of API key validity
# Useful for demos or offline development
FORCE_MOCK=false

# Flask session secret — change this before any non-local deployment
SECRET_KEY=change-this-in-production

# SQLite database file path (created automatically on first run)
DB_PATH=wanderlens.db
```

The application is fully functional without a Skyscanner key. In that case, it automatically serves a set of realistic mock flight results covering a range of prices, airlines, and stop combinations, so the UI and sorting logic can still be demonstrated.

## File Reference

### `app.py`

The application entry point. Initialises Flask, SQLAlchemy, and Flask-Login. Registers all blueprints with their URL prefixes (`/`, `/auth`, `/input`, `/search`, `/api`). Creates the database schema on first run via `db.create_all()`. Also defines the profile editing, password change, and account deletion routes, all of which require an active login session and operate directly on the current user record.

### `models.py`

Defines three SQLAlchemy models backed by a local SQLite database:

- **User** — stores the username, a scrypt-hashed password, accumulated reward points, and travel preferences (style tags, accessibility needs, budget tier). Provides computed properties for experience and video counts.
- **Experience** — a written travel review linked to a user and a destination city, with a 1–5 star rating.
- **Video** — a video record with file paths for both the original upload and the VR-converted version. The `vr_ready` boolean indicates whether the ffmpeg conversion completed successfully.

### `blueprints/auth_bp.py`

Handles `/login`, `/register`, and `/logout`. Passwords are hashed with Werkzeug's `scrypt` method. On successful login, Flask-Login issues a persistent session cookie. The blueprint validates minimum length for both username (3 characters) and password (6 characters) before creating an account.

### `blueprints/input_bp.py`

Manages the community contribution section of the platform:

- `POST /input/experience` — saves a travel review and awards 20 points to the author.
- `POST /input/upload-video` — saves the uploaded video to disk, invokes ffmpeg to produce a side-by-side VR version using a lens-correction filter, and awards 50 points. If ffmpeg is not available, the video is saved without conversion and the route still returns a success response.
- `POST /input/upload-photo` — saves a photo and awards 30 points.
- `GET /input/leaderboard` — returns the top 10 users ordered by points.
- `DELETE` routes allow users to remove their own content; points are deducted proportionally.

### `blueprints/search_bp.py`

Contains the Gemini AI travel planner:

- Detects user intent using keyword matching across English, Catalan, and Spanish without making an additional API call.
- If a destination or sufficient travel intent is detected, constructs a structured prompt and calls Gemini, requesting a strict JSON response with destination coordinates, accessibility notes, sustainability tips, and an IATA airport code used to auto-populate the flight search panel.
- Retries up to three times with progressively lower temperature if the model returns malformed or unparseable JSON.
- Falls back to a friendly conversational message if all retries fail.
- If intent is unclear, calls Gemini with a shorter clarification prompt to ask the user for missing information.

### `blueprints/api_bp.py`

Acts as a server-side proxy so that API keys are never exposed to the browser:

- `GET /api/flights` — calls the Skyscanner Indicative Search API and returns results sorted by price, direct-first, or date.
- `GET /api/hotels` and `GET /api/cars` — delegate to `skyscanner_client.py`.
- `GET /api/map-points` — geocodes a city via Nominatim, then queries the Overpass API for nearby points of interest filtered by type (museum, restaurant, park, etc.).
- `GET /api/community` — retrieves community-contributed experiences and videos for a given city from the local database.
- `GET /api/health` — reports the availability of all external dependencies.

### `blueprints/skyscanner_client.py`

Low-level Skyscanner API client. Constructs the request payload in the exact format required by the indicative search endpoint (`/v3/flights/indicative/search`), handles HTTP errors and timeouts, parses the `quotes` dictionary from the response, and returns a normalised list of flight objects that the rest of the application consumes. Falls back to structured mock data if no API key is present or if `FORCE_MOCK=true` is set in the environment.

## VR Panoramic Viewer — CodePen Integration

### Rationale

Mobile devices with limited RAM — particularly mid-range and low-end Android phones — cannot reliably render WebGL-based 360-degree panoramic video inline within the application. Embedding an A-Frame scene directly in the page caused memory allocation failures on devices with less than 3 GB of RAM, and produced no output at all on iOS due to Safari's hard limit on WebGL context memory.

To work around this, the VR panoramic viewer is hosted as a standalone CodePen. The user is redirected to the pen, which runs in the device's full browser context with unrestricted access to GPU memory. This approach works correctly on Android and on all desktop browsers. It is **not supported on iOS** regardless of the browser used, as this is a platform-level WebGL constraint that cannot be bypassed from a web page.

## Known Limitations

**VR viewer not available on iOS**
The 360-degree viewer does not function on iOS devices. Safari on iOS enforces a memory ceiling on WebGL contexts that prevents A-Frame and equivalent libraries from rendering spherical content. No web-based workaround exists; a native iOS application would be required to support this feature on that platform.

**VR viewer performance on low-RAM Android devices**
Devices with less than 3 GB of RAM may experience slow initial loading or reduced frame rates. Uploading content at 720p resolution rather than 1080p or higher significantly improves stability on such devices.

**ffmpeg not installed**
VR side-by-side video conversion requires ffmpeg to be present in the system PATH. If it is not found, the video is stored in its original format and the `vr_ready` flag is set to `false`. The rest of the application continues to operate normally.

**Skyscanner Indicative Search limitations**
The hackathon API key provides access to the Indicative Search endpoint (`/v3/flights/indicative/search`), which returns price estimates per calendar day rather than exact scheduled departure and arrival times. Precise flight scheduling data requires the Live Pricing endpoint (`/v3/flights/live/search/create`), which is available under a separate API tier not included in the hackathon access.

**Gemini response reliability**
Gemini occasionally returns malformed JSON on the first attempt, particularly for complex multi-day itineraries. The application retries up to three times with progressively lower temperature settings before falling back to a conversational message. A production deployment would benefit from response caching or a more deterministic post-processing step.

## Sustainability

WanderLens incorporates principles from the United Nations 2030 Sustainable Development Goals:

**SDG 10 — Reduced Inequalities.** Accessibility information is a first-class field in both user profiles and AI-generated itineraries. Users with mobility, visual, or auditory needs receive tailored stop recommendations and transport advice. Contributors who provide accessibility-specific reviews are awarded additional points to recognise the value of this information to other travellers.

**SDG 12 — Responsible Consumption and Production.** Every generated itinerary includes sustainability tips and cultural respect guidelines. The AI planner is instructed to flag destinations affected by over-tourism and to suggest less-visited alternatives. The community points system incentivises the contribution of original, locally-informed content over reliance on mainstream travel aggregators.
